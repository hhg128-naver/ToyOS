# Stage 7: 유저 모드(User Mode) 및 시스템 콜(System Call) 구현 완료 보고서

## 1. 개요 (Overview)
본 문서는 ToyOS 개발 7단계인 '유저 모드 전환 및 시스템 콜 인프라 구축' 과정에서 발생한 주요 기술적 난관과 이를 해결하기 위해 적용된 기술적 조치들을 상세히 기록합니다.

## 2. 발생 이슈 및 해결 방안 (Issues & Solutions)

### ISSUE 1: 비캐노니컬 주소에 의한 General Protection Fault (#GP)
- **현상**: 유저 모드 진입을 시도하자마자 `Exception 13` 발생.
- **원인**: 
  - x86_64 아키텍처의 엄격한 주소 규정인 **캐노니컬 주소(Canonical Address)** 위반.
  - 유저 스택 주소를 `0x00007FFFFFFFF000`으로 설정하고 4KB를 더해 최상단 주소를 계산한 결과 `0x0000800000000000`이 됨. 
  - 비트 47번이 1임에도 상위 비트들이 모두 1이 아니어서 CPU가 이를 유효하지 않은 주소로 판단하고 예외 발생.
- **해결**: 유저 스택 주소를 안전한 캐노니컬 범위인 `0x00007FFFFFFF0000`으로 하향 조정.

### ISSUE 2: GDT 레이아웃 및 `sysret` 명령어 호환성 문제
- **현상**: 유저 모드 주소 수정 후에도 동일한 `Exception 13` 발생 (Error Code: 0x62 등).
- **원인**: 
  - `sysret` 명령어는 GDT 세그먼트 배치가 특정 순서(User Data 바로 다음에 User Code)로 되어 있을 것을 강제함.
  - `MSR_STAR` 레지스터에 설정된 베이스 선택자 값이 CPU의 내부 계산 방식과 미묘하게 어긋남.
  - 일부 하드웨어에서 GDT 엔트리의 `Accessed` 비트가 미리 설정되지 않으면 예외가 발생하는 특성 확인.
- **해결**: 
  - `MSR_STAR`의 유저 베이스 선택자를 `0x10`(Index 2)으로 표준화.
  - GDT 유저 세그먼트의 Access 비트에서 `Accessed` 비트(bit 0)를 명시적으로 활성화 (`0xFA` -> `0xFB`, `0xF2` -> `0xF3`).

### ISSUE 3: 페이징 권한 전파(Permission Propagation) 누락
- **현상**: 유저 모드 코드 실행 중 커널 영역 함수 호출이나 데이터 참조 시 예외 발생.
- **원인**: 
  - 커널 초기화 시 Identity Mapping(0~1GB) 영역에 `PAGE_USER` 비트가 없었음.
  - 최하위 페이지 테이블(PT)에만 유저 권한을 줘도, 상위 테이블(PML4, PDPT, PD) 중 하나라도 Supervisor 권한이면 접근이 차단됨.
- **해결**: 
  - `VMM_Init`에서 초기 매핑 영역 전체에 `PAGE_USER` 권한 부여 (테스트 환경용).
  - `VMM_MapPage` 함수를 수정하여 하위 페이지가 유저 권한을 요구할 경우, 상위 테이블들에도 자동으로 `PAGE_USER` 및 `PAGE_WRITABLE` 권한을 누적 적용하도록 로직 개선.

### ISSUE 4: 시스템 콜 리턴 메커니즘의 불안정성
- **현상**: `sysretq` 명령어 실행 후 유저 모드 복귀 과정에서 스택 전환 실패.
- **원인**: `sysret`은 `RSP`를 자동으로 복원하지 않으며, `RCX`(RIP)와 `R11`(RFLAGS) 레지스터 관리가 매우 까다로움.
- **해결**: 
  - `SyscallEntry`에서 복귀 방식을 `sysretq`에서 **`iretq`**로 전면 교체.
  - 스택에 `SS`, `RSP`, `RFLAGS`, `CS`, `RIP`를 수동으로 쌓아 iretq 프레임을 구성함으로써, 권한 전환과 스택 전환을 CPU가 원자적으로(Atomically) 처리하도록 보장.

## 3. 최종 구현 성과
1. **유저 모드 진입 성공**: Ring 0에서 Ring 3로의 안전한 권한 전환 확인.
2. **시스템 콜 인터페이스 완성**: `syscall` 명령어 -> `SyscallEntry`(asm) -> `SyscallHandler`(C) -> 서비스 처리의 전체 사이클 구축.
3. **안정적인 컨텍스트 스위칭**: 멀티태스킹 스케줄러 내에서 유저 태스크와 커널 태스크가 충돌 없이 병렬 실행됨.
4. **표준 C 라이브러리(Newlib) 연동 준비**: `printf` 등 유저 모드에서 사용할 서비스의 백엔드(syscalls.c) 연결 완료.

## 4. 향후 과제 (Next Steps)
- **유저 공간 독립**: 현재 커널과 공유 중인 주소 공간을 분리하여 태스크별 독립된 페이지 테이블 구축.
- **ELF 로더**: 커널 내장 함수가 아닌, 실제 디스크의 실행 파일을 읽어 실행하는 기능 추가.
- **추가 시스템 콜 확장**: 파일 입출력(VFS), 프로세스 관리(fork, exec) 등 필수 시스템 콜 구현.
